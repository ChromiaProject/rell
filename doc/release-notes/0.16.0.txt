RELEASE NOTES 0.16.0 (2026-05-28)

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
1. Compiler: Resolved runtime model (RR_*)

A new resolved runtime model now sits between compilation and execution. After compilation finishes, the compiler
produces a self-contained, immutable RR_App:

    Source code  ->  compiler  ->  RR_App  ->  interpreter

Previously, the compiler output could not be serialized or sent outside the JVM process. This blocked cross-process IR
transport, alternative backends, and any tooling that needs to consume a Rell program without running the compiler
itself.

The new RR_ model is plain immutable data with no compiler dependencies and no hidden behavior on nodes. Execution
logic lives entirely in the interpreter rather than on the tree itself.

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
2. Compiler: Module layout of rell-base

The old monolithic rell-base has been split into sub-modules so the serialized IR format is defined independently of
compilation and execution:

    ┌──────────────────────┬────────────────────────────────────────────────────────────────┐
    │ Sub-module           │ Purpose                                                        │
    ├──────────────────────┼────────────────────────────────────────────────────────────────┤
    │ utils                │ Shared utilities, no Rell dependencies                         │
    │ rr-tree              │ RR_ data classes (depends only on utils)                       │
    │ rr-serialization     │ Serialization/deserialization of RR_App (depends on rr-tree)   │
    │ frontend             │ Compiler, AST, R_ model, resolve()                             │
    │ runtime-core         │ Runtime values, types, SQL generation, standard library        │
    │ runtime-interpreter  │ Tree-walking interpreter over RR_App (depends on runtime-core) │
    └──────────────────────┴────────────────────────────────────────────────────────────────┘

A consumer that only needs to deserialize an RR_App pulls in rr-tree and rr-serialization, without the frontend or the
runtime. The frontend/runtime split is enforced at the Gradle module level: the runtime cannot accidentally import
compiler internals, and the compiler cannot depend on runtime-only classes. The runtime is itself split into
runtime-core (values, types, SQL, stdlib) and runtime-interpreter (the RR_App tree-walking executor): an alternative
backend — for example, a future native compiler — can reuse the value model and standard library without pulling in
the interpreter.

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
3. API: compileApp() returns RR_App

Public API entry points (compileApp() and related) now return RR_App instead of R_App. RR_App is much simpler to
stabilize and evolve as a public surface. Remaining compiler and runtime types are hidden behind @InternalRellApi.

This is a breaking change for integrators that consumed R_App directly.

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
4. API: Serialization of compiled Rell programs

The rr-serialization module serializes RR_App to a compact, language-neutral binary format (FlatBuffers). A compiled
Rell program can be produced on one machine and executed on another without re-running the compiler, including by
consumers not written in Kotlin/Java.

Invariant:

    interpret(compile(code)) == interpret(deserialize(serialize(compile(code))))

Anything the interpreter needs must live on the RR tree itself. If a round-trip through serialization changes
behavior, it is a bug. The invariant is enforced in the build: every ./gradlew check runs the entire existing test
suite twice - once against RR_App produced directly by the compiler, and once against RR_App that went through a
serialize/deserialize round-trip.

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
5. Runtime: Security of serialized RR_App

A crafted RR_App binary cannot make the interpreter do anything that valid Rell source code cannot. The RR tree is
data, not executable code: it can only express operations the compiler itself emits, and no others. A malformed binary
may crash the interpreter or cause denial-of-service at the app level, but cannot escalate privileges or bypass the
standard library. In the Postchain deployment model, that impact is confined to the app that shipped the bad binary.

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
6. Runtime: SQL generation rebuilt on jOOQ

Every PostgreSQL statement the runtime emits — entity reads, INSERT/UPDATE/DELETE, schema initialisation
(CREATE TABLE, ALTER TABLE, CREATE INDEX), snapshots, meta-data updates — is now built from a structured
jOOQ query AST instead of concatenated strings. Identifiers go through jOOQ's PostgreSQL renderer, which
removes a class of identifier-quoting risks the previous emitter relied on convention to avoid.

Two latent schema-init bugs surfaced and were fixed by the rewrite. Adding an attribute that had both a
foreign-key and a size-check constraint silently dropped one of the two. @max_size(N) with N above
2,147,483,647 was truncated to a negative value, producing a CHECK that rejected every INSERT. Both
behaviours now work as documented.

The exact SQL text written to PostgreSQL logs and pg_dump differs in whitespace, parenthesisation, and
keyword casing — the queries are semantically equivalent and produce identical result sets. A new
regression test (rell-base/src/test/kotlin/sql/SqlEmissionTest.kt) pins the emitted SQL for a
representative corpus so any future drift is caught at CI time.

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
7. Runtime: Global system functions moved out of per-chain initializeDB

Initialization of Rell's global SQL system functions has been extracted from the per-chain initializeDB path
into a new RellGlobalStorageInitializer registered as a Postchain GlobalStorageInitializer service. Previously,
these functions were (re)installed during every chain's database initialization, which under concurrent chain
startup could deadlock on shared catalog locks (POS-2115).

With this change, global system functions are created once per storage during global initialization, and per-chain
initializeDB no longer touches them. Existing deployments are migrated transparently; no user action is required.

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
8. API: IdeApi.getModuleInfo overload that recognises folder-based test modules

IdeApi.getModuleInfo(C_SourcePath, S_RellFile) could only inspect the file's own header to decide whether the
file belonged to a test module. For a folder module:

    tests/module.rell      // @test module;
    tests/test_one.rell    // function test_foo() {}
    tests/test_two.rell    // function test_bar() {}

only module.rell carries the @test module; header, so the IDE plugin was unable to detect that
test_one.rell and test_two.rell are part of a test module and did not show the run-test gutter icon for
their test functions (RELL-252).

A new overload that takes the source directory as well is now available:

    IdeApi.getModuleInfo(sourceDir: C_SourceDir, path: C_SourcePath, ast: S_RellFile): IdeModuleInfo?

When the file has no module header, the new overload looks at the sibling module.rell and inherits the test
flag from it. The original two-argument overload is unchanged for backwards compatibility — IDE integrations
that need correct test detection on folder-module members should switch to the new overload.

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
9. Compiler: ANTLR is the new mainline parser

The Rell frontend parser is now built from a hand-tuned ANTLR4 grammar instead of the better-parse Kotlin combinator
library. Every production parse entry point runs the new pipeline; better-parse is no longer on the production code
path.

The pipeline uses an SLL fast path with bail-out on prediction conflict, and falls back to a full LL parse with error
recovery only on inputs that actually contain syntax errors. For valid (hotpath) programs this is several times faster
than the previous combinator-based parser. The LL fallback collects every recoverable error in a file rather than
bailing on the first one, so IDE diagnostics can surface the full list together with a (possibly partial) AST.

The previous parser is kept in tree as a reference implementation, reachable only through a test-only entry point. A
differential test gate parses every Rell snippet in the test corpus through both pipelines and compares the resulting
ASTs; any future divergence fails the build.

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
10. Tooling: BNF grammar facility removed for IntelliJ plugin consumers

The previously published helper for generating an IntelliJ GrammarKit BNF from the Rell grammar has been
removed, along with its supporting parser-introspection surface. It walked the better-parse combinator tree;
with the better-parse parser retired, that helper was deleted.

No replacement BNF generator ships from rell-base or rell-toolbox. The ANTLR grammar shipped with the
frontend is the single source of truth for Rell syntax; downstream plugins should consume it directly — for
example via antlr/antlr4-intellij-adaptor (https://github.com/antlr/antlr4-intellij-adaptor), which wraps an
ANTLR parser as an IntelliJ PsiParser without an intermediate BNF.

This is a breaking change for the IntelliJ plugin and any other consumer that built a BNF off the old transformer.

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
11. Runtime: GraalVM Truffle peer backend

A second runtime backend sits alongside the tree-walking interpreter. Both consume the same RR_App, the same
standard library, and the same database connection; they differ only in dispatch. The Truffle backend translates each
user function/operation/query body to a Truffle AST and lets GraalVM's partial evaluator specialize hot paths.
Sub-trees that don't yet have hand-written Truffle nodes fall through to the tree-walker, so semantics are identical
from day one.

The tree-walker remains the only supported backend for deployed applications. The Truffle backend is exercised
internally by the test matrix and microbenchmarks but is not yet ready for production use. Performance on real-world
Rell workloads is mixed — early measurements show the two backends within the same order of magnitude on the FT4
library and on dapp-level workloads (mna-blockchain), with Truffle ahead on some shapes and behind on others. We are
not publishing comparative numbers until the implementation has stabilized.

For experimentation, the backend can be selected with the JVM system property rell.execution.backend, which accepts
interpreter (the default) or truffle. To run the chr CLI with the Truffle backend, pass the property through
JAVA_ARGS:

    JAVA_ARGS="-Drell.execution.backend=truffle" chr run <args>

This switch is experimental and for testing only — it is not a supported configuration for
deployed applications, and the property may change or be removed without notice.

The CI suite now runs the entire rell-base test family three times — interpreter, interpreter + RR-serialization
round-trip, and Truffle — and requires 100% results on all three. Differences between the tree-walker and
Truffle are Truffle bugs; the tree-walker is the canonical reference.

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
12. Tooling: rell-dokka-plugin no longer depends on Dokka

The doc-site generator has been rewritten as a pure HTML generator. It walks the Rell L_/R_ model directly
(stdlib via Lib_Rell + Lib_RellTest, source projects via RellApiBaseInternal.compileApp) and produces the
output tree with kotlinx.html, commonmark, and a self-contained stylesheet. All Dokka dependencies and the unshading
workaround that backed them are gone.

The CLI surface is unchanged: `--source/--target/--modules/--name/--system/--styles/--assets/--includes/
--source-link/--filtered-modules/--additional-modules` continue to work as before. The two public API classes that
downstream tools depend on keep the same constructor and method signatures, so existing chromia-cli build (and any
other caller) compile against the new plugin without edits. The rendered output is a new look but the Dokka path
layout is preserved.
