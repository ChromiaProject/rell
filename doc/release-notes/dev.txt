1. Language: Native functions

One can define a native function in Rell this way:

    @native function f(a: integer, b: text): boolean;

The function can be called from Rell code, like a normal function.

For each Rell module which defines native functions, there must be a Java (or Kotlin) class implementing them. Such
classes are specified in the blockchain configuration at the path gtx.rell.native, one class per module. In XML, it
would look like:

    <dict>
        ...
        <entry key="gtx">
            <dict>
                ...
                <entry key="rell">
                    <dict>
                        ...
                        <entry key = "native">
                            <dict>
                                <entry key = "my_module">
                                    <string>com.domain.pack.MyNativeModule</string>
        ...
    </dict>

In Kotlin, the class MyNativeModule may look like:

    class MyNativeModule(val env: RellNativeEnvironment) {
        // @native function f(x: integer): integer;
        fun f(x: Long): Long {
            return x * x
        }
    }

Rules for native module classes:
1. The class must be public.
2. It must have a public constructor either without parameters, or with a single parameter of type
   RellNativeEnvironment.
3. Must be a public instance (non-static) function for each native function defined in the corresponding Rell module.
   The name of the native function must be exactly the same as the name of the Rell function.
4. The number of parameters and parameter types must match (see a table below).

The interface RellNativeEnvironment provides some information about the blockchain, which may be of use:

    interface RellNativeEnvironment {
        val config: Gtv
        val blockchainRid: BlockchainRid
    }

Kotlin types needed to implement native modules, like RellNativeEnvironment, are defined in the Maven artifact:

    net.postchain.rell:rell-api-native:<RELL_VERSION>

It has to be added to dependencies of the project where native functions are to be implemented.

Allowed parameter and return types:

┌────────────────────┬───────────────────────┐
│ Rell type          │ Kotlin type           │
├────────────────────┼───────────────────────┤
│ big_integer        │ java.math.BigInteger  │
│ boolean            │ Boolean               │
│ byte_array         │ ByteArray             │
│ decimal            │ java.math.BigDecimal  │
│ gtv                │ net.postchain.gtv.Gtv │
│ integer            │ Long                  │
│ rowid              │ Long                  │
│ text               │ String                │
│ unit (return type) │ Unit                  │
│ T? (nullable)      │ T?                    │
└────────────────────┴───────────────────────┘

Note. Implementation of a native function has to be deterministic, i.e. always return the same result for same parameter
values. Rell can't control this, so it's the native function developer's responsibility.

2. Runtime: Execute guard block during operation validation

When a Postchain node receives a transaction, it validates the transaction before executing it. Nodes may not write to
the database during the validation step, though are permitted to read from it. They may read and write to the database
during the execution step.

Rell operation guard blocks are run during operation validation, in addition to also being run in operation execution.
Only variable declarations are allowed to occur in an operation before a guard block (variable assignment is NOT allowed
before a guard). Thus, the following is legal:

    operation foo(y: integer) {
        val x: integer;
        guard {
            ...
        }
        ...
    }

However, the following is illegal, since x is assigned a value before the guard:

    operation foo(y: integer) {
        val x = y + 1;
        guard {
            ...
        }
        ...
    }

Since guard blocks are run both in the validation and execution steps, any log() or print() (or similar) calls that
occur in the guard are executed twice, and will result in duplicate logged/printed lines.

Guard blocks are forbidden from making database modifications (since they need to run during validation), though may
query the database. The op_context struct is accessible within the guard block during validation, but the
op_context.emit_event() method is not allowed (runtime error during validation).

In Postchain, the validation step is known as checkCorrectness(), and the execution step is known as apply().
The checkCorrectness() step is called from Postchain's BaseTransactionQueue.enqueue()
(net.postchain.base.data.BaseTransactionQueue), when a transaction is sent to a node (rather than when it's executed
i.e. the apply() step).

3. Library: GTV big_integer JSON serialization

GtvBigInteger values are now always supported in JSON serialization and are serialized as JSON numbers
(not strings). This means gtv.to_json() can now serialize big integers without any special flags or parameters.

Examples:

    123L.to_gtv().to_json() // Returns json[123]
    123L.to_gtv().to_json() // Returns json[123]
    123456789012345678901234567890L.to_gtv().to_json() // Returns json[123456789012345678901234567890]

IMPORTANT: Backward compatibility note for deserialization:

When deserializing JSON containing big integers back to GTV using gtv.from_json(), the behavior depends on
the magnitude of the number:

- Small big integers (within the range of a standard JSON number/integer) will be deserialized as GtvInteger,
  not GtvBigInteger. This causes type information loss.

- Large big integers (exceeding standard integer range) cannot be deserialized and will result in a runtime error.

Examples:

    // Small big_integer loses type information when round-tripped
    val serialized = 123L.to_gtv().to_json();
    gtv.from_json(serialized) // Returns gtv[123] (a regular integer, not big_integer)

    // Large big_integer cannot be deserialized
    val large = 123456789012345678901234567890L.to_gtv().to_json();
    gtv.from_json(large) // Runtime error
