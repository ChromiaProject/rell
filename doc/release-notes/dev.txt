1. Language: Annotations @singular and @compound for operations

Postchain v3.34.0 introduced two new operation types:

- A compound operation. This is an operation that cannot 'stand alone' in a transaction. A transaction that contains a
  compound operation without any non-compound operations is invalid.
- A single-per-transaction operation. This is an operation that cannot occur more than once per transaction.

Rell operations can now be marked as compound with the @compound annotation, and as single-per-transaction with the
@singular annotation.

Examples:

    @compound operation finalize(...) { ... }
    @singular operation credit_account(...) { ... }

2. Tests: Function-level @test annotation

Where one would previously write:

    function test_foo(...) { ... }

One can now instead write:

    @test function foo(...) { ... }

Functions with the @test annotation must be defined in test modules, and may not take any parameters. If these
restrictions are violated, the result is a compiler error. If such conditions are violated with tests defined by their
name, they are silently ignored. It is therefore strongly recommended to use the @test annotation instead of defining
tests by their name.

The @test annotation may only be used on functions that are not otherwise annotated with @extendable or @extend, or have
no abstract or override modifiers.

3. Library: Functions text.match_groups() and text.match_named_groups()

New functions to retrieve capturing groups from a regular expression match:

- text.match_groups(regex: text): list<text>?
- text.match_named_groups(regex: text): map<text, text>?

Both functions match the entire text value against the specified regular expression (as opposed to searching for a
match), returning null if the match fails.

text.match_groups() returns a list<text> (if the match succeeds) containing the text matched to each match group, the
zeroth of which is the entire matched text.

text.match_named_groups() returns a map<text, text> (if the match succeeds) containing match group names as keys, and
their matching text as values. Text matched to unnamed groups is not returned (use text.match_groups() instead for
these).

Match groups in a regular expression are defined by any parentheses that the regular expression contains, and the groups
are ordered by the position of their opening parentheses. For example, the regular expression (X(Y))(Z) contains 3
matching groups, which are:

- (X(Y))
- (Y)
- (Z)

Named groups are match groups in a regular expression for which a name is specified. The match groups in the above
example (X(Y))(Z) could be assigned names in the following manner:

    (?<x>X(?<y>Y))(?<z>Z)

This is an equivalent regular expression, but the text matched by each group can be referenced by the group's name.

Both functions support non-capturing groups (notated (?:X), where X is a regular expression), but do not include
such groups in the returned list or map.

Named groups that match the empty string with the '?' and '*' quantifiers are included in the map returned by
text.match_named_groups(), but named groups within unmatched alternatives are not.

Examples for text.match_groups():

- 'johnsmith@chromaway.com'.match_groups('([a-z]+)@([a-z]+[.][a-z]+)') returns
    ['johnsmith@chromaway.com', 'johnsmith', 'chromaway.com'].
- 'XYZ'.match_groups('(X(Y))(Z)') returns ['XYZ', 'XY', 'Y', 'Z'].
- 'XYZ'.match_groups('((X(Y))(Z))') returns ['XYZ', 'XYZ' ,'XY', 'Y', 'Z'].
- 'X'.match_groups('(X)|(Y)') returns ['X', 'X', ''] (the third group (Y) matches nothing, hence the empty string).
- 'XYZ'.match_groups('(?<x>X(?<y>Y))(?<z>Z)') returns ['XYZ', 'XY', 'Y', 'Z'].

Examples for text.match_named_groups():

- 'XYZ'.match_named_groups('(?<x>X(?<y>Y))(?<z>Z)') returns ['x': 'XY', 'y': 'Y', 'z': 'Z'].
- 'XYZ'.match_named_groups('(?<x>X(Y))(Z)') returns ['x': 'XY'].
- 'johnsmith@chromaway.com'.match_groups('(?<user>[a-z]+)@(?<domain>[a-z]+[.][a-z]+)') returns
    ['user': 'johnsmith', 'domain': 'chromaway.com'].
- 'XYZ'.match_named_groups('(X(Y))(Z)') returns map<text, text>().
- 'X'.match_named_groups('(?<x>X)|(?<y>Y)') returns ['x': 'X'].
- 'X'.match_named_groups('(?<x>X)(?<y>Y?)') returns ['x': 'X', 'y': ''].

4. API: Functions to check the validity of Rell tokens

New API object RellApiTokenizer, with functions:

    isIdentifier(s: String): Boolean
    isInteger(s: String, allowSign: Boolean = true): Boolean
    isBigInteger(s: String, allowSign: Boolean = true): Boolean
    isDecimal(s: String, allowSign: Boolean = true): Boolean
    isText(s: String): Boolean
    isByteArray(s: String): Boolean

Each function takes a String and checks if it is a valid Rell token of the named type, using the same rules as the Rell
compiler. The numeric types take an optional parameter that controls whether a sign (+ or -) is allowed to precede the
number (defaults to true).

In each case, if a String that is a valid Rell token is passed, true is returned, and false otherwise. Neither
whitespace nor comments are allowed (false will be returned in either case).

Examples:

    RellApiTokenizer.isIdentifier("abc") returns true
    RellApiTokenizer.isIdentifier("2nd_value") returns false (identifiers cannot start with a numeric digit)

    RellApiTokenizer.isInteger("1234") returns true
    RellApiTokenizer.isInteger("9223372036854775808") returns false (out of range)

    RellApiTokenizer.isDecimal(".0") returns true
    RellApiTokenizer.isDecimal("+55.77e-5", allowSign = false) returns false (signed, but sign not allowed)

    RellApiTokenizer.isBigInteger("-1234567L") returns true
    RellApiTokenizer.isBigInteger("+ 0x1234abcdL", allowSign = true) returns false (illegal space after '+')

    RellApiTokenizer.isText("'Hello world!'") returns true
    RellApiTokenizer.isText("Hello world!") returns false (no quote marks)

    RellApiTokenizer.isByteArray("x\"\"") returns true
    RellApiTokenizer.isByteArray("x\'00\'") returns true

5. API: Line endings normalization during compilation of app to GTV

When compiling Rell source files to GTV format, all line endings in the source code are now automatically normalized to
Unix-style line endings (\n). This ensures consistent behavior across different platforms and operating systems.

This change affects the GTV compilation process performed by RellApiCompile.compileGtv(...) and ensures that the
compiled GTV output contains normalized source code regardless of the original file's line-ending format.
